Final Reflection

Final Reflection on the Implementation of the Website

13 June 2025

A final reflection on the design and technical implementation of my portfolio website. This post explores how my user interface (UI) and user experience (UX) decisions influenced my folder structure, semantic HTML, metadata, CSS styling, and JavaScript functionality. It also covers challenges I faced with mobile responsiveness and code organisation, and what I learned about building accessible and maintainable websites.

Abstract

This reflection explores the design and development process of my portfolio website, with a focus on how user interface (UI) and user experience (UX) choices impacted the technical implementation. It discusses how my folder structure, semantic HTML, CSS, and JavaScript features evolved throughout the semester. It also covers the challenges I faced, such as text styling and mobile responsiveness, and the improvements I plan to make going forward.

Folder Structure

Hierarchical Organisation

My folder structure follows the layout of my website. Each main section - such as blog posts, essays, animations, and galleries - has its own folder. Assets like images, stylesheets, and scripts are also separated into their own folders. This helped keep things organised and made it easier to find and update files later.

Separation of Concerns

I separated HTML, CSS, and JavaScript into different files. This made the project cleaner and easier to maintain:

  • HTML handles the content and structure
  • CSS takes care of styling and layout
  • JavaScript adds functionality and interactivity

At first, it was hard to get used to this structure, especially when I was updating styles across pages. But in the long run, it made my code more flexible and easier to debug.

Modularity and Naming

I used simple but meaningful class names like .content-card, .gallery-image, and .lightbox. I also reused layout classes and section structures across different pages. This made it easier to scale the site and keep styles consistent.

Semantic Markup

Structure and Meaning

I used semantic HTML elements like <header>, <main>, <nav>, and <article> to clearly communicate the purpose of each section of content.

Expressing Content Clearly

Headings are arranged in a logical hierarchy. Lists, buttons, and links are clearly labelled. This helps both users and screen readers understand the layout and content.

Improving Accessibility

Using semantic markup improved:

  • Screen reader usability (clear reading order, headings)
  • Keyboard navigation (logical tab flow)
  • Search engine optimization (structured content)
  • Mobile accessibility (better behaviour on assistive devices)

Metadata

Browser Support and SEO

Each page includes metadata like the character set, page description, and viewport settings for mobile responsiveness. This helps browsers render the content properly and also improves visibility on search engines.

Useful Metadata for Specific Content

Pages like essays or blog posts could be improved even further with more detailed metadata, like publication dates or keywords. While I didn’t include full structured data yet, it’s something I plan to explore later for better SEO and content description.

CSS Styling

Style Consistency

One of the biggest challenges I faced was maintaining consistent text styling. I found myself repeating the same font sizes, colours, and spacing rules for headings and paragraphs across different selectors. I now realise I could’ve created shared utility classes or used CSS variables to keep things more consistent and maintainable.

Visual Design and Effects

Some custom CSS features include:

  • Profile photo hover: changes to the original image when hovered
  • Name hover: a waving emoji animates when you hover over my name
  • Blog carousel: scrolls one card at a time within a styled frame

Mobile Responsiveness

The site is responsive thanks to Flexbox and media queries, but it’s not exactly how I imagined it in my mobile wireframes. Some sections feel squished or need better padding/margins on smaller screens. I’d like to keep improving the mobile layout by refining breakpoints and simplifying the structure.

JavaScript Functionality

Interactive Features

I used JavaScript to add functionality without relying on inline code. Key features include:

  • Typed name animation on the homepage
  • Blog post carousel with navigation arrows
  • Image and video lightboxes in fullscreen modals
  • Keyboard support (ESC to close, arrow keys to navigate)

Writing Reusable Code

I avoided inline JavaScript and wrote all scripts in external files. This kept the code clean and easier to reuse. For example, the modal and carousel logic is used in multiple places across the site.

Async and Dynamic Elements

Even without APIs or form handlers, the site includes dynamic behaviour such as:

  • Modals that load new content based on clicks
  • Lightboxes that adapt based on how many media items are available
  • Typing effect that animates over time

Contact and Links

I chose not to include a custom contact form. Instead, I used a simple and effective solution: a Contact Me button that links to my email, Instagram, and other social platforms. This method doesn’t require JavaScript or APIs and still gives visitors a clear way to connect with me.

Challenges and Improvements

  • Repeated CSS for text: I now realise I could’ve used shared classes or CSS variables instead of repeating styles.
  • Mobile layout issues: My responsive design didn’t fully match my mobile wireframes. I plan to simplify it and test more on devices.
  • Learning better CSS structure: I’ve started building a proper global style system instead of fixing things case-by-case.
  • Code consistency: I’ve gotten better at writing clean, reusable code and avoiding bad habits like overly specific selectors.

Conclusion

Making this website helped me understand how UI and UX choices affect every part of the code — from file structure and HTML to how I write my JavaScript. I learned how important it is to separate concerns, use semantic elements, and keep styles consistent. Even though I faced challenges with text styles and responsive design, I now have a solid foundation I can build on. I’m proud of the result and excited to keep improving the site as I grow as a developer and designer.
